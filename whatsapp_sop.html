WhatsApp UTM Tracking — SOP-ZAP-001

This workflow captures and stores UTM (Urchin Tracking Module) parameters from WhatsApp traffic via Zapier. It ensures that each unique visitor session — identified by a client ID (CID) — is recorded exactly once in the tracking table, preventing duplicate entries and maintaining clean attribution data.

SOP ID: SOP-ZAP-001  |  Status: Active  |  Platform: Zapier + Zapier Tables  |  Version: 1.1

1. Scope

This SOP applies to:

  • Marketing and analytics teams responsible for campaign attribution
  • Operations and automation teams managing Zapier workflows
  • Any stakeholder reviewing WhatsApp campaign performance data

2. Tools & Systems

ToolRole
ZapierWorkflow automation engine
Zapier WebhookReceives the incoming UTM payload (trigger)
Zapier Tables — WA_UTM_ClicksStores one record per unique CID

3. Trigger

The workflow is triggered by an incoming Webhook (HTTP POST). This webhook fires whenever a user arrives at a tracked page via a WhatsApp link containing UTM parameters.

Expected webhook payload fields:

FieldDescription
cidUnique client/session identifier
utm_sourceTraffic source (e.g. whatsapp, google)
utm_mediumMarketing medium (e.g. cpc, organic)
utm_campaignCampaign name
utm_termPaid search keyword (optional)
utm_contentAd variant (optional)
page_urlLanding page URL
referrerReferring URL
timestampISO 8601 timestamp of the session

4. Workflow Steps

The automation runs through four sequential steps each time the webhook fires:

1
Receive Webhook (Trigger)

The workflow starts when the Zapier webhook endpoint receives a POST request. The payload is parsed and all UTM fields are made available to subsequent steps. No action is taken at this point.

2
Search for Existing Record

Zapier searches the WA_UTM_Clicks table in Zapier Tables for the incoming CID.

  • Lookup field: cid (mapped to column f1)
  • Match type: exact match
  • If no match found → workflow continues to Step 3
  • If match found → filter in Step 3 will stop the workflow

Note: The search is configured to succeed on miss — it does not fail if no record is found; it continues normally.

3
Filter — Skip Duplicates

Condition: record_id from Step 2 does not exist.

  • If record_id is empty (no match) → proceed to Step 4, create new entry
  • If record_id has a value (match found) → workflow stops, no duplicate written
⚠ Deduplication guarantee: Each unique CID is stored only once, regardless of how many times the webhook fires for the same session.
4
Create Record in Zapier Tables

If no duplicate is found, a new row is written to WA_UTM_Clicks with all 9 UTM/page/timestamp fields mapped to the correct table columns.

5. Process Flow

Webhook fires ↓ Step 1: Parse payload (CID + UTM fields) ↓ Step 2: Search WA_UTM_Clicks for CID ↓ Step 3: Is record_id empty? YES → Step 4: Write new row NO → STOP (duplicate suppressed)

6. Testing Checklist

  • Send a test webhook payload with a new CID → confirm a record is created in WA_UTM_Clicks
  • Send the same CID a second time → confirm no duplicate record is created
  • Verify all 9 UTM/page/timestamp fields are correctly mapped to the right columns
  • Monitor Zapier task history for 48 hours and confirm no failures

7. Maintenance & Ownership

Review this Zap whenever:

  • WhatsApp campaign UTM parameter structure changes
  • The WA_UTM_Clicks table schema is modified
  • Zapier updates the Webhook or Tables integration

Any changes to this workflow must be reflected in an updated version of this document.
