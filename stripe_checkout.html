Stripe Checkout Sistemi

ITIN ödeme sayfasının uçtan uca dokümantasyonu — mimari, bakım, yeni paket ekleme ve aynı sistemin yeni bir marka için sıfırdan kurulumu.

← SOP Library
Document: SOP-STRIPE-001 v1.0 Owner: CRM Specialist (Emel Gedik) Approved by: Dorukhan Korkut Status: Active Last Updated: Nisan 2026
Önemli not — eski plan ile fiili build farklı. SOP Library'deki "ITIN STRIPE CHECKOUT" planlama dokümanı, Stripe Checkout API (hosted checkout sayfasına redirect) üzerinden anlatılmıştı. Fiili build farklıdır: custom HTML formu içinde Stripe Elements + PaymentIntent API kullanılıyor. Bu SOP, fiili build'in dokümantasyonudur. Eski planlama dokümanı referans olarak tutulabilir ama operasyonel kararlarda bu SOP esas alınmalıdır.

1. Amaç

Bu SOP, ITIN ödeme sayfasının teknik altyapısını dokümante eder. Ödeme akışının nasıl çalıştığı, hangi servislerin birbirine bağlandığı, gün içinde ne tür hatalarla karşılaşılabileceği, yeni paket eklenmesi gerektiğinde atılacak adımlar ve aynı sistemin yeni bir marka için kurulması istendiğinde izlenecek yol burada anlatılır.

Bu doküman, sistemi devralacak veya bakımına dahil olacak her ekip üyesi için tek başvuru kaynağıdır.

2. Kapsam

  • Frontend: Cloudflare Pages üzerinde host edilen tek sayfalık ödeme formu (public/index.html).
  • Backend: Cloudflare Pages Functions (serverless) — üç endpoint: /create-payment-intent, /payment-success, /webhook.
  • WordPress entegrasyonu: her paket için ayrı WordPress sayfası, ödeme formu iframe olarak gömülü.
  • Stripe hesabı: Oguz Law Stripe (live) ve test ortamı.
  • HubSpot entegrasyonu: Forms API submit + opsiyonel CRM API contact update.

Bu SOP kapsam dışındadır: Stripe ürün/fiyat tasarımı (pazarlama kararı), WordPress sayfa içerikleri (içerik ekibi), Stripe ödeme uyuşmazlık (dispute) süreci.

3. Sistem Mimarisi

3.1 Stack

  • Hosting: Cloudflare Pages (statik sayfa) + Cloudflare Pages Functions (serverless API endpoint'leri). Build adımı yok, output dizini public/.
  • Ödeme: Stripe Elements (kart girişi) + Stripe PaymentIntent API (ödeme oluşturma).
  • CRM: HubSpot Forms API (tüm submit'ler) + HubSpot CRM API (token varsa contact property update).
  • Site: WordPress (paket sayfaları, iframe wrapper).

4. Uçtan Uca Ödeme Akışı

  1. Kullanıcı WordPress paket sayfasını açar. Her paket için ayrı bir sayfa var (örn. itin.com/standard-itin). Sayfada Cloudflare ödeme formu iframe olarak gömülü, paket query param ile geçiriliyor: ?pkg=itin.
  2. Frontend açılışta UTM ve tracking yakalar. URL'den okur, localStorage ile birleştirir, hubspotutk cookie'sini de alır. Yakalanan alanlar: utm_source/medium/campaign/term/content/id, gclid, fbclid, msclkid, referrer, landing_page, first/last touch URL ve tarih, page_history, page_views, device, browser_lang.
  3. Form doldurulur ve submit edilir. Frontend, POST /create-payment-intent çağırır. Body: email, ad-soyad, telefon, adres, premiumSelected, pkg, tracking bloğu.
  4. Backend PaymentIntent hazırlar.
    • Stripe API'den base price çeker (gerekirse premium price'ı da).
    • customers/search ile email'e göre müşteri arar. Bulursa update eder, bulamazsa yeni oluşturur.
    • PaymentIntent oluşturur. UTM, tracking, müşteri bilgisi, IP (Cloudflare cf-connecting-ip header'ından), paket bilgisi — hepsi PI metadata'sına yazılır.
    • page_history, Stripe metadata 500 karakter limiti nedeniyle path-only string'e sıkıştırılır.
    • client_secret frontend'e döner.
  5. Frontend Stripe ile ödemeyi onaylar. stripe.confirmCardPayment(clientSecret, { payment_method: { card, billing_details } }). Kart bilgisi Stripe'a doğrudan gider, sunucumuza dokunmaz (PCI scope dışında kalırız).
  6. Ödeme başarılıysa frontend POST /payment-success'a istek atar. Burada PaymentIntent ID'si, contact data, paket, tracking, success URL gönderilir.
  7. Backend ödemeyi server-side doğrular ve HubSpot'a yazar.
    • PaymentIntent'i tekrar çekip status === 'succeeded' kontrolü.
    • HubSpot Forms API submit (Portal 44184085, Form bc104a87-0a64-42e3-abf7-1c9192975a9e). UTM'ler native HubSpot field name'leriyle, hutk context'e ekli, payment info hidden field olarak.
    • HUBSPOT_ACCESS_TOKEN set ise CRM API üzerinden contact'ı email'le bulup property'leri PATCH eder: payment_status, payment_amount, payment_id, last_payment_date, package_type.
  8. Frontend thank-you sayfasına yönlendirir. https://itin.com/thank-you/#<params> — hash fragment, query değil (GA ve iframe parent için kasıtlı). Iframe içindeyse window.top.location.href kullanılır, üst pencere yönlendirilir.

5.3 Frontend display config

Frontend'deki PACKAGES nesnesi backend ile aynı key'leri kullanır ama sadece görsel bilgi içerir: name (başlık), sub (alt başlık), base (gösterim fiyatı), upsellDesc (premium upsell açıklaması). Fiili tahsilat tutarı her zaman Stripe'tan çekilir — frontend'deki sayı sadece görsel.

Neden iki yerde tutuyoruz? Backend Stripe API'yi konuşur, fiyatı her ödeme öncesi gerçek zamanlı çeker (manipülasyon riski yok). Frontend ise sayfa render olur olmaz fiyatı göstermek zorunda — bu yüzden hızlı bir hardcoded değere ihtiyacı var. İki yerde tutarlı kalmaları operasyonel sorumluluktur.

6. WordPress Entegrasyonu

Mevcut pattern: her paket için ayrı bir WordPress sayfası açılır. Sayfada HTML widget içine iframe gömülür. Iframe, Cloudflare Pages URL'ine ?pkg=<key> query param ile yönlendirilir.

<div style="max-width:700px;margin:0 auto;">
  <iframe
    src="https://stripe-hubspot2.pages.dev/?pkg=itin"
    style="width:100%;border:none;min-height:100vh;"
    id="payment-frame"></iframe>
</div>

<script>
window.addEventListener('message', function(e) {
  if (e.data && e.data.type === 'resize') {
    document.getElementById('payment-frame').style.height = e.data.height + 'px';
  }
});
</script>

Iframe yüksekliği postMessage ile ödeme formundan parent pencereye gönderilir, otomatik resize olur. Sayfa template'i Full Width olmalı (sidebar yok).

7. Environment Variables

Cloudflare Pages → Settings → Environment Variables → Production altında tanımlanır. Değişiklik sonrası en son deployment'i Retry etmek zorunlu — yoksa yeni env var'lar pickup edilmez.

STRIPE_SECRET_KEYEvetsk_live_… — Stripe Dashboard → Developers → API Keys'ten alınır. Yoksa /create-payment-intent 500 döner.
HUBSPOT_ACCESS_TOKENHayırSet ise CRM contact property update yapılır. Yoksa sadece Forms API submit gider.

Stripe publishable key (pk_live_…) public/index.html içinde hardcoded. Bu güvenlik açığı değildir (publishable key zaten halka açıktır), ama test/live arasında geçişte unutulan alanlardan biridir. Bkz. §10.5.

8. UTM ve HubSpot Entegrasyonu

8.1 UTM yakalama

Frontend captureUtms() fonksiyonu sayfa açılır açılmaz çalışır:

  1. URL query string'inden UTM ve tracking parametrelerini okur.
  2. localStorage ile birleştirir (boş kalan alanları doldurur, dolu olanları saklar).
  3. document.referrer'ı yakalar.
  4. hubspotutk cookie'sini okur.
  5. Device tipi ve browser dilini ekler.

8.2 HubSpot Forms API submit

Her başarılı ödemede tetiklenir:

HUBSPOT_ACCESS_TOKEN set ise: contact email ile aranır, bulunursa property'leri PATCH'lenir. Bu sayede HubSpot'ta contact kaydında payment property'leri güncel kalır. Token yoksa bu adım atlanır, sadece form submit kalır.

9. Bakım ve İşletme

9.1 Deploy

GitHub'a main branch'e push otomatik Cloudflare Pages deployment'ını tetikler. Build adımı yok, deployment ortalama 30-60 saniye. Hotfix'ler doğrudan main'e push edilebilir — staging branch yok.

9.2 Loglar

Cloudflare Dashboard → Workers & Pages → ilgili proje → Functions → Real-time Logs. Tüm console.log/console.error çıktıları burada görünür. Hata debugging için create-payment-intent.js içindeki debugStep değişkeni ne aşamada hata aldığını söyler (örn. "DEBUG [customer-search]: …").

9.3 Bilinen gotchas

  1. Webhook signature verification yok. webhook.js, Stripe'tan gelen payload'ı parse edip log'luyor ama stripe-signature header'ını doğrulamıyor. Şu an webhook spoof edilebilir. Production'da signature verify edilmeli.
  2. Webhook handler aslında iş yapmıyor. Sadece log'luyor. Asıl HubSpot submit /payment-success üzerinden frontend tetiklemesiyle gidiyor. Risk: Kullanıcı tarayıcısı /payment-success isteğini atmadan kapatırsa ödeme Stripe'ta görünür ama HubSpot'ta kayıt oluşmaz. Çözüm: webhook'u gerçek handler'a çevirmek (uzun vadeli iş).
  3. HubSpot submit fail-silent. Try/catch ile yutuluyor, sadece console.error. Frontend'e hata dönmüyor — kullanıcı thank-you sayfasına gidiyor ama CRM'de kayıt olmamış olabilir. Loglardan kontrol edilmeli.
  4. Stripe customer search eventual consistency. Yeni oluşmuş bir customer hemen aranınca bulunamayabilir, sistem yeniden create eder → duplicate Stripe customer oluşur. Çözüm gelecek versiyona kalmıştır; şimdilik aynı email'le çok hızlı arka arkaya iki ödeme yapılırsa Stripe Dashboard'da temizlik gerekebilir.
  5. Test/Live geçişi manuel. Otomatik switch yok. Test'e geçmek için: public/index.html'de stripe_pk değerini pk_test_… ile değiştir + Cloudflare env var'larında STRIPE_SECRET_KEY'i sk_test_… yap + deployment retry. Live'a dönüşte aynı işlemi tersine yap.
  6. Stripe metadata 500 karakter limiti. page_history kırpılıyor ama uzun referrer/URL'lerde Stripe API 400 dönebilir. Yeni alan eklerken .substring(0, 500) uygulanmalı.
  7. Iframe'in yüksekliği parent'a postMessage ile gidiyor. Parent script'i çalışmazsa scroll bar oluşur. WordPress sayfası kurulurken script'in eklendiğinden emin ol.

10. Yeni Paket Ekleme — Adım Adım

Mevcut sisteme yeni bir paket eklemek istendiğinde izlenecek sıra. Mevcut marka için (yani aynı Stripe ve HubSpot hesabı altında) geçerlidir; yeni marka için bkz. §11.

  1. Stripe Dashboard'da product ve price oluştur. Stripe → Products → Add product. Tek seferlik ödeme (one-time) seç, fiyatı gir, kaydet. Oluşan Price ID'yi kopyala (price_… ile başlar). Test mod ve live mod için ayrı Price ID'ler vardır — ikisini de oluştur.
  2. Backend'i güncelle: functions/create-payment-intent.js. Dosyanın en üstündeki PACKAGES nesnesine yeni satır ekle: 
    'yeni-paket-key': {
    price: 'price_XXXXXXXXXX',
    name: 'Görünür Paket Adı'
}
  3. Frontend'i güncelle: public/index.html. PACKAGES nesnesine aynı key ile yeni satır ekle: 
    'yeni-paket-key': {
    name: 'Görünür Paket Adı',
    sub: 'Alt başlık',
    base: 297,
    upsellDesc: 'Premium upsell açıklaması.'
}
    Önemli: Backend ve frontend key'leri tıpatıp aynı olmalı. Tutarsızlık olursa frontend default itin'e döner.
  4. Commit ve push. Git üzerinden main branch'e push. Cloudflare Pages otomatik deploy eder, yaklaşık 30-60 saniye.
  5. WordPress'te yeni paket sayfası oluştur. Yeni sayfa aç, HTML widget ekle, §6'daki iframe kodunu kopyala, ?pkg=yeni-paket-key olarak güncelle. Template Full Width yap. Yayınla.
  6. Test modunda doğrula. Önce test moduna geç (§9.3 madde 5). Yeni paket sayfasını aç, test kartı ile ödeme dene. Kontrol listesi:
    • Ödeme formunda doğru paket adı, fiyat, premium açıklaması görünüyor mu?
    • Test kartı ile ödeme tamamlanıyor mu?
    • Stripe Dashboard'ta PaymentIntent oluşmuş ve metadata'da paket bilgisi doğru mu?
    • HubSpot'a contact düşüyor mu? (Form submission'ları kontrol et.)
    • Thank-you sayfasına yönlendirme oluyor mu?
  7. Live moda geri dön. Test pk/sk değerlerini geri al, deployment retry et.
  8. Live ortamda son bir kontrol yap. Mümkünse en düşük tutarlı bir test paketi üzerinden gerçek bir ödeme dene (sonradan refund), çünkü test mod ve live mod arasında bazen farklılıklar çıkabilir.

11. Yeni Marka için Sıfırdan Kurulum — Adım Adım

Aynı sistemin başka bir marka (örn. yeni bir LLC, ayrı bir hizmet hattı) için kurulması gerektiğinde izlenecek yol. Bu, mevcut sistemin klonlanıp yeni Stripe ve HubSpot hesaplarına bağlanmasıdır.

11.1 Hazırlık

  1. Stripe hesabını hazırla. Yeni marka için ayrı bir Stripe hesabı varsa o kullanılır; yoksa mevcut hesap altında ürün eklenebilir (multi-brand karar). Karar netleşene kadar başlamayın.
  2. HubSpot brand kararı. Aynı HubSpot portal'ı içinde ayrı brand olarak mı kurulacak, yoksa ayrı portal'a mı gidecek? Bkz. SOP Library → "HubSpot Internal Kurallar". Genelde aynı portal'da brand olarak ayrılır.
  3. Domain ve URL planı. Ödeme sayfası hangi subdomain altında olacak? (örn. pay.yenimarka.com). WordPress siteye bağlanacak mı?

11.2 Stripe ürünlerini ve fiyatlarını oluştur

  1. Stripe Dashboard → Products → her paket için product + price oluştur.
  2. Premium add-on varsa onu da ayrı price olarak oluştur.
  3. Hem test hem live için ayrı Price ID'ler kaydet — iki ortam ayrı yönetilir.
  4. Tüm Price ID'leri bir dokümana not et (örn. Google Sheets) — repo güncellenirken bu dokümandan kopyalanacak.

11.3 HubSpot tarafını hazırla

  1. HubSpot → Marketing → Forms → yeni form oluştur. Form ödeme akışına özel: firstname, lastname, email, phone + UTM alanları + payment hidden field'ları (payment_status, payment_amount, payment_id, package_type).
  2. Form'un brand'ini doğru seç (HubSpot Internal Kurallar SOP'una göre).
  3. Portal ID ve Form GUID değerlerini al — repo'da bunlar değişecek.
  4. (Opsiyonel) HubSpot Private App üzerinden access token oluştur — CRM contact property update'i için. Custom property olarak payment_status, payment_amount, payment_id, last_payment_date, package_type ekle (yoksa).

11.4 Repo'yu klonla ve özelleştir

  1. Mevcut stripe-hubspot-main repo'sunu yeni bir repo'ya kopyala (örn. stripe-hubspot-yenimarka). Fork değil, yeni repo — birbirinden bağımsız ilerleyecekler.
  2. functions/create-payment-intent.js içinde:
    • PACKAGES nesnesini yeni markanın paketleriyle baştan yaz.
    • PREMIUM_PRICE ve PREMIUM_TEST sabitlerini yeni Price ID'lerle güncelle (premium add-on yoksa kaldır).
  3. functions/payment-success.js içinde:
    • HS_PORTAL_ID ve HS_FORM_ID değerlerini yeni HubSpot bilgileriyle değiştir.
    • HubSpot custom property isimleri farklıysa updateHubSpotContact içindeki property map'i güncelle.
  4. public/index.html içinde:
    • PACKAGES nesnesini yeni markanın paketleriyle baştan yaz (backend'le birebir aynı key'ler).
    • CFG.stripe_pk değerini yeni markanın pk_live_… publishable key'iyle değiştir.
    • Logo'yu public/logo.png içinde değiştir.
    • Başlıklar, renkler, success URL (https://yenimarka.com/thank-you/#…) gibi marka-spesifik alanları güncelle.
  5. DEPLOY-GUIDE.md'i güncelle — yeni domain, yeni env var değerleri, yeni Stripe hesabı referansları. Bu doküman başka birisi devraldığında ilk okuyacağı şey.

11.5 Cloudflare Pages projesi oluştur

  1. Cloudflare Dashboard → Workers & Pages → Create → Pages → Connect to Git → yeni repo'yu seç.
  2. Build command boş, build output dir public.
  3. Save and Deploy. İlk deploy <proje-adi>.pages.dev URL'ini verir.
  4. Settings → Environment Variables → Production:
    • STRIPE_SECRET_KEY: yeni markanın sk_live_…
    • HUBSPOT_ACCESS_TOKEN: (opsiyonel)
    • ALLOWED_ORIGIN: https://yenimarka.com
  5. Deployments → en son deployment'ı Retry — env var'lar pickup olur.
  6. (Önerilir) Custom domain ekle: pay.yenimarka.com CNAME → <proje-adi>.pages.dev.

11.6 Stripe webhook bağla

  1. Stripe Dashboard → Developers → Webhooks → Add endpoint.
  2. URL: https://pay.yenimarka.com/webhook (veya pages.dev URL'i).
  3. Event'ler: payment_intent.succeeded, payment_intent.payment_failed.
  4. Hatırlatma: webhook handler şu an signature verify etmiyor (§9.3 madde 1). Yeni marka kurulumu fırsattır — bu handler'ı düzgün implemente et: webhook secret'ı env var'a koy, signature doğrula, gerçek HubSpot submit'i webhook'tan tetikle (frontend bağımlılığını kaldır).

11.7 WordPress entegrasyonu

  1. Yeni markanın WordPress sitesinde her paket için ayrı sayfa aç.
  2. Her sayfada §6'daki iframe kodunu kullan, src'yi yeni Cloudflare URL'ine çevir, ?pkg=<key> uygun olsun.
  3. Sayfa template'i Full Width.
  4. Yayınlamadan önce iframe yüksekliğinin postMessage ile düzgün resize olduğunu kontrol et.

11.8 Test ve canlıya alma

  1. İlk deploy'ı önce test mod ile yap: pk_test_… + sk_test_….
  2. Her paket sayfasını test kartıyla test et (§9.4).
  3. Stripe Dashboard ve HubSpot'ta verinin düzgün düştüğünü kontrol et — özellikle UTM ve metadata.
  4. Test geçtikten sonra live anahtarlara geç, deployment retry.
  5. Live ortamda en düşük tutarlı paketle 1 gerçek ödeme yap, refund et — production sanity check.
  6. Test mode onayı için Dorukhan Korkut'a not düş (mevcut sistemde olduğu gibi).

12. Zapier Entegrasyonu (Legacy)

Zapier'da iki Zap bu ödeme akışına bağlı: Zap #15 (Stripe → completed checkout session) ve Zap #18 (Stripe → new payment). Bkz. Zapier Active Automations — Master Reference.

Mevcut custom checkout (PaymentIntent) sistemi bu Zap'ları doğrudan tetiklemez. Ancak eski Stripe Payment Link'ler hâlâ aktif — birisi eski bir link üzerinden ödeme yaparsa Zap #15 ve #18 devreye girer. Bu yüzden iki Zap da kapatılmamalı, legacy fallback olarak duruyor.

Yeni paket eklerken eski Payment Link'lerle bir alakası yok — onları görmezden gelebilirsin. Yeni marka kurulumunda da Zapier tarafına dokunmaya gerek yok, çünkü yeni sistemde HubSpot submit doğrudan /payment-success üzerinden gidiyor.
