Stripe Checkout System

End-to-end documentation for the ITIN payment system: architecture, maintenance, adding a new package, and rebuilding the same system from scratch for a new brand.

Important — the original plan and the actual build differ. The "ITIN STRIPE CHECKOUT" planning document in the SOP Library described a Stripe Checkout API flow (redirecting users to a hosted Stripe checkout page). The actual build is different: a custom HTML form with Stripe Elements + the PaymentIntent API. This SOP documents the actual build. The earlier planning document remains as historical reference, but operational decisions should be based on this SOP.

1. Purpose

This SOP documents the technical infrastructure of the ITIN payment page. It explains how the payment flow works, which services connect to which, what kinds of errors can occur day-to-day, the steps to follow when adding a new package, and how to set up an equivalent system for a new brand from scratch.

This document is the single source of reference for any team member taking over the system or contributing to its maintenance.

2. Scope

  • Frontend: single-page payment form hosted on Cloudflare Pages (public/index.html).
  • Backend: Cloudflare Pages Functions (serverless) — three endpoints: /create-payment-intent, /payment-success, /webhook.
  • WordPress integration: a separate WordPress page per package, with the payment form embedded as an iframe.
  • Stripe account: Oguz Law Stripe (live + test).
  • HubSpot integration: Forms API submit + optional CRM API contact update.

Out of scope for this SOP: Stripe product/price design (a marketing decision), WordPress page content (content team), Stripe dispute handling.

3. System Architecture

3.1 Stack

  • Hosting: Cloudflare Pages (static page) + Cloudflare Pages Functions (serverless API endpoints). No build step, output directory public/.
  • Payments: Stripe Elements (card input) + Stripe PaymentIntent API (payment creation).
  • CRM: HubSpot Forms API (every successful payment) + HubSpot CRM API (contact property update when access token is set).
  • Site: WordPress (package pages, iframe wrapper).

4. End-to-End Payment Flow

  1. The user opens a WordPress package page. Each package has its own page (e.g. itin.com/standard-itin). The Cloudflare payment form is embedded as an iframe; the package is passed as a query parameter: ?pkg=itin.
  2. The frontend captures UTM and tracking data on load. It reads from the URL, merges with localStorage, and reads the hubspotutk cookie. Captured fields: utm_source/medium/campaign/term/content/id, gclid, fbclid, msclkid, referrer, landing_page, first/last touch URL and date, page_history, page_views, device, browser_lang.
  3. The form is filled in and submitted. The frontend calls POST /create-payment-intent. Body: email, first/last name, phone, address, premiumSelected, pkg, tracking block.
  4. Backend prepares the PaymentIntent.
    • Fetches the base price from Stripe (and the premium price if applicable).
    • Searches Stripe customers by email via customers/search. Updates if found, creates a new one if not.
    • Creates a PaymentIntent. UTM, tracking, customer info, IP (from Cloudflare's cf-connecting-ip header), package info — all attached as PI metadata.
    • page_history is compressed to a path-only string due to the 500-character Stripe metadata value limit.
    • Returns client_secret to the frontend.
  5. Frontend confirms the payment with Stripe. stripe.confirmCardPayment(clientSecret, { payment_method: { card, billing_details } }). Card details go directly to Stripe and never touch our server (which keeps us out of PCI scope).
  6. On success, the frontend calls POST /payment-success. Body: PaymentIntent ID, contact data, package, tracking, success URL.
  7. Backend verifies the payment server-side and writes to HubSpot.
    • Re-fetches the PaymentIntent and checks status === 'succeeded'.
    • Submits to HubSpot Forms API (Portal 44184085, Form bc104a87-0a64-42e3-abf7-1c9192975a9e). UTMs go in as native HubSpot fields, hutk goes in the context, payment info goes as hidden fields.
    • If HUBSPOT_ACCESS_TOKEN is set: looks up the contact by email via the CRM API and PATCHes properties: payment_status, payment_amount, payment_id, last_payment_date, package_type.
  8. Frontend redirects to the thank-you page. https://itin.com/thank-you/#<params> — note the hash fragment, not query string (intentional, for GA and iframe parent compatibility). If running inside an iframe, window.top.location.href is used to redirect the parent window.

5.3 Frontend display config

The frontend's PACKAGES object uses the same keys as the backend, but only contains display info: name (title), sub (subtitle), base (display price), upsellDesc (premium upsell description). The actual amount charged is always pulled live from Stripe — the number on the frontend is just for display.

Why is it duplicated in two places? The backend talks to Stripe and fetches the price live before each charge (no manipulation risk). The frontend has to display a price the moment the page renders, so it needs a fast hardcoded value. Keeping the two consistent is an operational responsibility.

6. WordPress Integration

Current pattern: a separate WordPress page is created for each package. An HTML widget is added to the page, containing an iframe pointing at the Cloudflare Pages URL with the ?pkg=<key> query parameter.

<div style="max-width:700px;margin:0 auto;">
  <iframe
    src="https://stripe-hubspot2.pages.dev/?pkg=itin"
    style="width:100%;border:none;min-height:100vh;"
    id="payment-frame"></iframe>
</div>

<script>
window.addEventListener('message', function(e) {
  if (e.data && e.data.type === 'resize') {
    document.getElementById('payment-frame').style.height = e.data.height + 'px';
  }
});
</script>

The iframe height is sent from the payment form to the parent window via postMessage, so it auto-resizes to fit content. The page template should be set to Full Width (no sidebar).

Defined under Cloudflare Pages → Settings → Environment Variables → Production. After any change, the latest deployment must be retried manually — otherwise the new variables will not be picked up.

STRIPE_SECRET_KEYYessk_live_… — from Stripe Dashboard → Developers → API Keys. If missing, /create-payment-intent returns 500.
HUBSPOT_ACCESS_TOKENNoIf set, the CRM contact properties are PATCHed after each payment. If absent, only the Forms API submission runs.

The Stripe publishable key (pk_live_…) is hardcoded inside public/index.html. This is not a security issue (publishable keys are meant to be public), but it is one of the fields easy to forget when switching between test and live modes. See §9.3.

8. UTM and HubSpot Integration

8.1 UTM capture

The frontend's captureUtms() function runs immediately on page load:

  1. Reads UTM and tracking parameters from the URL query string.
  2. Merges with localStorage (fills empty fields, preserves existing ones).
  3. Captures document.referrer.
  4. Reads the hubspotutk cookie.
  5. Adds device type and browser language.

8.2 HubSpot Forms API submit

Triggered after every successful payment:

Portal ID
44184085
Form ID
bc104a87-0a64-42e3-abf7-1c9192975a9e
Endpoint
https://api.hsforms.com/submissions/v3/integration/submit/<portalId>/<formId>

Submitted fields: firstname, lastname, email, phone, the five UTMs, gclid/fbclid/msclkid/utm_id, referrer, landing_page, payment_status, payment_amount, payment_id, package_type. The hutk is included in the context — this is what lets HubSpot tie the submission back to the visitor's existing browser session.

8.3 HubSpot CRM API contact update (optional)

If HUBSPOT_ACCESS_TOKEN is set, the contact is looked up by email and patched with payment-related properties. This keeps the contact record up to date with the latest payment. If the token is missing, this step is skipped and only the form submission persists.

9. Maintenance and Operations

9.1 Deploy

A push to the main branch on GitHub automatically triggers a Cloudflare Pages deployment. No build step, deployments take ~30–60 seconds. Hotfixes can go straight to main — there is no staging branch.

9.2 Logs

Cloudflare Dashboard → Workers & Pages → the project → Functions → Real-time Logs. All console.log and console.error output appears here. For debugging, the debugStep variable inside create-payment-intent.js tells you which stage the error happened at (e.g. "DEBUG [customer-search]: …").

9.3 Known gotchas

  1. Webhook signature verification is not implemented. webhook.js parses and logs the Stripe payload but does not verify the stripe-signature header. The webhook is currently spoofable. Signature verification should be added before relying on the webhook in production.
  2. The webhook handler does no real work. It only logs. The actual HubSpot submit happens via /payment-success, triggered by the frontend. Risk: if the user closes their browser before the /payment-success request fires, the payment will appear in Stripe but no record will land in HubSpot. Long-term fix: move the HubSpot submit into the webhook.
  3. HubSpot submission fails silently. Errors are swallowed by a try/catch and only printed to console.error. The user is still redirected to the thank-you page even if the CRM never received the data. Logs must be monitored to catch this.
  4. Stripe customer search is eventually consistent. A customer just created may not be findable by customers/search immediately, so the system creates a duplicate. Two payments by the same email in rapid succession can produce duplicate Stripe customers. Manual cleanup in the Stripe Dashboard may be needed; a retry/dedupe layer is on the backlog.
  5. Test/Live mode switching is manual. There is no automatic switch. To switch to test: change stripe_pk in public/index.html to pk_test_…, change the STRIPE_SECRET_KEY env var to sk_test_…, and retry the deployment. Reverse the same steps to go back live.
  6. Stripe metadata 500-character limit per value. page_history is already trimmed, but long referrer/URL values may still cause Stripe to return 400. When adding new metadata fields, always apply .substring(0, 500).
  7. Iframe height relies on postMessage. If the parent script does not run (e.g. it was forgotten when the WordPress page was set up), a scrollbar appears inside the iframe. Always confirm the script is in place when creating a new package page.

10. Adding a New Package — Step by Step

Use this section when adding a package to the existing system (same Stripe and HubSpot accounts). For setting up a brand-new instance, see §11.

  1. Create the product and price in Stripe Dashboard. Stripe → Products → Add product. Choose one-time payment, enter the price, save. Copy the resulting Price ID (starts with price_…). Test mode and live mode have separate Price IDs — create both.
  2. Update the backend: functions/create-payment-intent.js. Add a new entry to the PACKAGES object at the top of the file: 
    'new-pkg-key': {
    price: 'price_XXXXXXXXXX',
    name: 'Display Name'
}
  3. Update the frontend: public/index.html. Add an entry to the PACKAGES object with the same key: 
    'new-pkg-key': {
    name: 'Display Name',
    sub: 'Subtitle',
    base: 297,
    upsellDesc: 'Premium upsell description.'
}
    Important: the backend and frontend keys must match exactly. If they don't, the frontend silently falls back to itin.
  4. Commit and push. Push to main. Cloudflare Pages auto-deploys, ~30–60 seconds.
  5. Create the WordPress page. Open a new page, add an HTML widget, paste the iframe code from §6, and update ?pkg=new-pkg-key. Set the page template to Full Width. Publish.
  6. Verify in test mode. Switch to test mode first (§9.3 item 5). Open the new package page and try a payment with a test card. Checklist:
    • Does the form display the correct package name, price, and premium description?
    • Does the test card payment complete?
    • Is the PaymentIntent created in Stripe Dashboard with the correct package metadata?
    • Does the contact land in HubSpot? (Check Form Submissions.)
    • Does the redirect to the thank-you page work?
  7. Switch back to live mode. Restore the live pk and sk values, retry the deployment.
  8. Run a final live check. If possible, run one real payment on the cheapest test package and refund it afterwards — production sanity check, since test and live modes occasionally diverge.

11. Setting Up the System for a New Brand — Step by Step

Use this section when the same payment system needs to be stood up for a different brand (e.g. a separate LLC or a new service line). This is essentially a clone of the existing system, rewired to a different Stripe and HubSpot setup.

11.1 Preparation

  1. Decide on the Stripe account. If the new brand has its own Stripe account, use it. Otherwise, products can be added under the existing account (a multi-brand call). Don't start until this is settled.
  2. Decide on the HubSpot brand. Will it be a new brand inside the existing HubSpot portal, or a separate portal entirely? See SOP Library → "HubSpot Internal Kurallar". Usually it goes under the existing portal as a separate brand.
  3. Plan the domain and URL. Which subdomain will host the payment page? (e.g. pay.newbrand.com.) Is it being attached to a WordPress site?

11.2 Create Stripe products and prices

  1. Stripe Dashboard → Products → create a product + price for each package.
  2. If there is a premium add-on, create it as a separate price.
  3. Create both test and live Price IDs — the two environments are managed separately.
  4. Record every Price ID in a working document (e.g. Google Sheets) — you will copy from this doc when updating the repo.

11.3 Prepare HubSpot

  1. HubSpot → Marketing → Forms → create a new form. The form should match the payment flow: firstname, lastname, email, phone + UTM fields + payment hidden fields (payment_status, payment_amount, payment_id, package_type).
  2. Make sure the form is assigned to the correct brand (per the HubSpot Internal Kurallar SOP).
  3. Note the Portal ID and Form GUID — these will be hardcoded into the cloned repo.
  4. (Optional) Create a HubSpot Private App and generate an access token, for the CRM contact property update. Make sure the custom properties exist on the Contact object: payment_status, payment_amount, payment_id, last_payment_date, package_type.

11.4 Clone and customize the repo

  1. Copy the existing stripe-hubspot-main repo to a new repo (e.g. stripe-hubspot-newbrand). Don't fork — make a clean copy. The two will evolve independently.
  2. In functions/create-payment-intent.js:
    • Rewrite the PACKAGES object with the new brand's packages.
    • Update the PREMIUM_PRICE and PREMIUM_TEST constants with the new Price IDs (or remove them if there is no premium add-on).
  3. In functions/payment-success.js:
    • Replace HS_PORTAL_ID and HS_FORM_ID with the new HubSpot values.
    • If the HubSpot custom property names differ, update the property map inside updateHubSpotContact.
  4. In public/index.html:
    • Rewrite the PACKAGES object with the new brand's packages (keys must match the backend exactly).
    • Replace CFG.stripe_pk with the new brand's pk_live_… publishable key.
    • Replace public/logo.png with the new brand's logo.
    • Update brand-specific fields: titles, colors, success URL (https://newbrand.com/thank-you/#…), etc.
  5. Update DEPLOY-GUIDE.md with the new domain, env var values, and Stripe account references. This is the first thing the next engineer will read when picking up the system.

11.5 Create the Cloudflare Pages project

  1. Cloudflare Dashboard → Workers & Pages → Create → Pages → Connect to Git → select the new repo.
  2. Build command: empty. Build output directory: public.
  3. Save and Deploy. The first deployment yields a <project-name>.pages.dev URL.
  4. Settings → Environment Variables → Production:
    • STRIPE_SECRET_KEY: the new brand's sk_live_…
    • HUBSPOT_ACCESS_TOKEN: optional
    • ALLOWED_ORIGIN: https://newbrand.com
  5. Deployments → retry the latest deployment so the env vars are picked up.
  6. (Recommended) Add a custom domain: pay.newbrand.com CNAME → <project-name>.pages.dev.

11.6 Wire up the Stripe webhook

  1. Stripe Dashboard → Developers → Webhooks → Add endpoint.
  2. URL: https://pay.newbrand.com/webhook (or the pages.dev URL).
  3. Events: payment_intent.succeeded, payment_intent.payment_failed.
  4. Reminder: the webhook handler does not currently verify signatures (§9.3 item 1). A new-brand setup is the perfect opportunity to fix this properly: store the webhook secret as an env var, verify the signature, and trigger the actual HubSpot submit from the webhook (removing the dependency on the frontend).

11.7 WordPress integration

  1. On the new brand's WordPress site, create one page per package.
  2. On each page, paste the iframe code from §6, point src at the new Cloudflare URL, and set ?pkg=<key> appropriately.
  3. Page template: Full Width.
  4. Before publishing, confirm the iframe height auto-resizes correctly via the postMessage handler.

11.8 Test and go live

  1. First deploy in test mode: pk_test_… + sk_test_….
  2. Test every package page with a test card (§9.4).
  3. Verify in Stripe Dashboard and HubSpot that the data lands correctly — especially UTMs and metadata.
  4. After tests pass, switch to live keys and retry the deployment.
  5. Run one real payment on the cheapest package in the live environment, then refund it — production sanity check.
  6. Get test mode approval from Dorukhan Korkut, following the same pattern as the existing system.

12. Zapier Integration (Legacy)

Two Zaps in Zapier are connected to the payment flow: Zap #15 (Stripe → completed checkout session) and Zap #18 (Stripe → new payment). See Zapier Active Automations — Master Reference.

The current custom checkout (PaymentIntent) flow does not trigger these Zaps directly. However, old Stripe Payment Links are still active — if someone pays through one of those legacy links, Zap #15 and #18 take over. Both Zaps must therefore be left enabled as a legacy fallback.

When adding a new package, the legacy Payment Links are unrelated — you can ignore them. For a new brand setup, no Zapier work is needed either, since the new system writes to HubSpot directly via /payment-success.

15. Improvement Backlog

Items the system runs without today, but that should be addressed over time:

  1. Implement webhook signature verification — currently spoofable.
  2. Move the HubSpot submit into the webhook — removes the risk of losing a record if the user closes their browser before /payment-success fires.
  3. Make the test/live switch driven by env vars — get the publishable key out of the hardcoded HTML.
  4. Add retry logic to the Stripe customer search — protect against duplicate-customer race conditions.
  5. Wire HubSpot submission failures into a monitoring channel (e.g. a Slack alert) — replace the current fail-silent behavior.
