Complete maintenance guide for E-Judge: a Node.js service that syncs immigration hearing data from the EOIR API into Google Sheets via Slack and HTTP.

E-Judge — EOIR Hearing Sync Service

Runtime: Node.js ≥ 18  |  Version: 1.0.0  |  Last updated: 2026-04-08  |  PM2 app name: e-judge

1. Overview

E-Judge is a Node.js service that synchronizes immigration hearing data from the EOIR Case-Access API into Google Sheets. It exposes an HTTP API (including a Slack slash command endpoint) that triggers a sync run, detects changes against the previous state stored in the master sheet, and writes the updated dataset back to Google Sheets.

High-Level Architecture

Slack / HTTP client | v Express API ──► syncService ──► caseService ──► EOIR API (src/index.js) (lock + flow) (HTTP + retry) | | | └──► sheetService ──► Google Sheets + Drive | └── /health, /sync, /slack/command, /master-sheet-info, /test/master-sheet

2. Runtime Environment

Requirements

  • Node.js 18+ (uses native global fetch / AbortController)
  • A Google Cloud service account with access to the target Drive folder
  • Network egress to case-access.eoir.justice.gov and Google APIs
  • ~300–500 MB RAM (PM2 restarts at 500 MB)

4. API Endpoints

MethodEndpointDescription
GET/healthLiveness check — returns {"status":"ok"} if process is alive
POST/syncTriggers a full EOIR sync run. Requires cookie in body.
POST/slack/commandSlack slash command endpoint (ACKs immediately, processes in background)
GET/master-sheet-infoReturns metadata about the master Google Sheet
GET/test/master-sheetReturns first 10 rows for quick inspection

Cookie Requirements

  • Minimum length: 20 characters
  • Rejected if it contains control characters (\x00–\x1f except tab/CR/LF)
  • Cookie validation is skipped only in offline mock mode (useTestSheet && mockHearingChanges)

Example API Calls

# Production sync curl -X POST http://localhost:3002/sync \ -H "Content-Type: application/json" \ -d '{"cookie":"<EOIR_COOKIE>","isProductionMode":true}' # Offline mock (no cookie needed) curl -X POST http://localhost:3002/sync \ -H "Content-Type: application/json" \ -d '{"useTestSheet":true,"mockHearingChanges":true}'

5. Sync Flow Details

  1. Lock acquisition — syncService uses a timestamp-based lock (LOCK_TIMEOUT_MS = 10 min). Stale locks are auto-released.
  2. Fetch hearings — caseService calls EOIR API with a 30 s timeout.
  3. Filter — only hearings within the next 12 months are kept; duplicates by ScheduleId are removed.
  4. Fetch details — parallel batches of 10 requests with up to 3 retries and exponential backoff (base 200 ms). 429 responses are respected.
  5. Read previous data — sheetService reads both individuals and others tabs (range A2:P).
  6. Diff & write — compares hearing date/time/judge/court, marks rows as NEW / UPDATED, preserves judge rates, notes, and assigned lawyer unless a change is detected in production mode.
  7. Trailing cleanup — rows beyond the new data (up to row 10,000) are cleared to prevent stale entries.
  8. Lock release — always executed in a finally block.

6. Monitoring & Logs

PM2 Commands

pm2 status # overview of all processes pm2 logs e-judge # tail combined logs pm2 logs e-judge --lines 200 # last 200 lines pm2 logs e-judge --err # errors only pm2 monit # interactive dashboard

Log Files

  • logs/out.log — stdout
  • logs/error.log — stderr
  • logs/combined.log — merged

Health Check

Use GET /health from an external monitor (UptimeRobot, Cloud Run healthcheck, etc.). It does not touch EOIR or Google APIs — a 200 proves the process is alive, not that downstream dependencies are reachable.

7. Routine Maintenance Schedule

CadenceTasks
WeeklyCheck pm2 status / Cloud Run revision health; scan logs/error.log for new error signatures; confirm the master sheet is receiving updates (check Drive modification timestamp)
MonthlyRotate / verify the EOIR cookie with the team; review Google Cloud quotas (Sheets API & Drive API); run npm outdated and plan dependency updates; archive or trim log files if they exceed a few hundred MB
QuarterlyRotate Google service-account key (steps below); review ALLOWED_USER_IDS against current team; re-validate Drive folder ID and service account access

Quarterly: Rotate the Google Service Account Key

  1. Create a new key in Google Cloud Console for the same service account
  2. Copy it to credentials/service-account.json on the server (or update Secret Manager for Cloud Run)
  3. Restart the service: pm2 restart e-judge
  4. Verify with curl /health and a small /sync run against the test sheet
  5. Delete the old key from Google Cloud Console

Dependency Upgrades

npm outdated npm audit # Minor/patch updates: npm update npm install --production pm2 restart e-judge # Always test googleapis upgrades against the test sheet before touching production.

8. Troubleshooting

SymptomCause & Resolution
Service won't startRun pm2 logs e-judge --err --lines 50. Verify credentials file exists and is valid JSON. Check port is not already bound: sudo lsof -i :3002. Confirm Node.js ≥ 18: node -v.
/sync returns "Senkronizasyon yapilacak veri bulunamadi"Invalid or expired EOIR cookie, or the account genuinely has no upcoming hearings. Verify by hitting EOIR directly with the same cookie.
/sync returns "Su anda baska bir senkronizasyon islemi yapiliyor"Another sync run is in progress. Wait. If stuck for more than 10 minutes, the lock will auto-clear on the next call.
Slack command times outThe endpoint ACKs immediately and processes in background. If timeouts persist, check whether Express is blocked (CPU/memory) or whether the inbound URL is reachable from Slack.
Google API 403 / 401Service account lost folder access. Re-share the Drive folder with the service-account email as Editor, or rotate the key (§7 Quarterly).
Data looks wrong in the sheetUse GET /master-sheet-info and GET /test/master-sheet to inspect state. Re-run /sync with useTestSheet=true before touching production.

9. Security Notes

  • The /sync endpoint has no authentication beyond cookie validation — protect it with a firewall, VPN, Cloud Run IAM, or reverse-proxy auth.
  • /slack/command relies on the ALLOWED_USER_IDS allow-list. Slack request signature verification is not yet implemented; adding X-Slack-Signature verification is recommended before public exposure.
  • EOIR cookies appear in request bodies — avoid logging them. The current code only logs length, which is safe.
  • The service-account key grants write access to the Drive folder — treat it as a high-severity secret.
