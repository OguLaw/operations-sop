WhatsApp UTM Tracking — Team Handoff

This document covers the full architecture of the two-Zap WhatsApp UTM tracking system: how it works, current readiness status, and what needs to happen before launch.

Priority: HIGH  |  Status: Configured, pending activation  |  Platform: Zapier + Wati

1. Executive Summary

We have built a two-Zap automated system in Zapier that tracks UTM attribution for inbound WhatsApp leads. The system captures the marketing source of a visitor before they send a WhatsApp message, then automatically applies that attribution data to their contact record in Wati when they first make contact.

Without this system, all WhatsApp leads appear as unattributed — making it impossible to determine which campaigns, ads, or channels are driving inbound enquiries.

2. System Architecture

The system consists of two Zapier workflows operating in sequence, using two Zapier Tables as shared data stores:

ComponentRole
Zap #1 (SOP-ZAP-001)Receives UTM parameters from the website via webhook when a visitor clicks a WhatsApp link. Stores UTMs + CID in WA_UTM_Clicks table.
Zap #2Fires when a new message arrives in Wati. Extracts the CID from the greeting, looks up UTM data in WA_UTM_Clicks, and writes attribution to the Contacts table and Wati contact attributes.
WA_UTM_Clicks tableStores one row per unique CID: UTM source, medium, campaign, term, content, landing page, referrer, timestamp.
Contacts tableStores attribution data tied to each Wati contact: CID, UTM fields, attribution timestamp, "already attributed" flag (f10).

3. End-to-End Flow

1. Visitor clicks WhatsApp button on website (link contains CID + UTM params) 2. Page script fires POST to Zapier webhook → Zap #1 stores UTMs in WA_UTM_Clicks 3. Visitor sends first WhatsApp message in Wati (greeting includes CID) 4. Zap #2 fires on new Wati message 5. Formatter step extracts CID from greeting text 6. Lookup: search WA_UTM_Clicks for that CID 7. If found: write UTM data to Contacts table + update Wati contact attributes 8. Set f10 = true (already attributed) → future messages from this contact are skipped

4. Zap #2 — Detailed Step Breakdown

Zap #2 runs every time a new message arrives in Wati and follows a branching logic:

StepActionDetail
TriggerNew message in WatiFires on every inbound message
Filter AIs this the first message?Stops if contact has already been attributed (f10 = true)
FormatterExtract CID from greetingUses text pattern matching to pull CID from the greeting string
LookupSearch WA_UTM_Clicks by CIDExact match on the cid column
Filter BWas a UTM record found?Stops if no matching CID exists (visitor arrived without UTM)
WriteUpdate Contacts tableWrites all UTM fields + CID + timestamp
WriteUpdate Wati contact attributesPushes utm_source, utm_campaign, etc. to Wati
WriteSet f10 = trueMarks contact as attributed — prevents re-attribution on future messages

6. Open Questions

  • How is the CID passed from the website into the WhatsApp greeting message — is it embedded in a pre-filled message URL or added by a script?
  • What is the exact format of the greeting that contains the CID? The Formatter step needs the pattern to extract it reliably.
  • Should returning contacts (f10 = true, already attributed) ever be re-attributed if they click a new ad later? Currently the system skips them.
  • Are there greeting languages beyond English, Turkish, and Spanish that should be accounted for?
  • Who owns the Wati account and can verify contact attributes are being written correctly?
