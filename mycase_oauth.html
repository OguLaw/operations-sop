MyCase OAuth Proxy — Installation & Operations

SOP ID: SOP-MYCASE-001  |  Version: 1.0  |  Date: March 2026  |  Audience: Engineering & DevOps

1. Purpose & Scope

This SOP provides step-by-step guidance for installing, configuring, operating, and maintaining the MyCase OAuth Proxy server. It covers the full lifecycle from initial environment setup through production deployment and routine operations.

This document applies to anyone who:

  • Sets up or clones the mycase-oauth repository for development or production
  • Configures environment variables and external service dependencies
  • Deploys, monitors, or troubleshoots the OAuth proxy server
  • Performs token management or administrative operations

2. Prerequisites

Required Knowledge

  • Node.js application development and npm package management
  • Linux/macOS command-line operations
  • OAuth 2.0 authentication flows
  • Redis server administration (basic)
  • Environment variable and secret management

Access Requirements

  • Source code repository (Git access)
  • MyCase Developer Portal account with application registration rights
  • Target server with sudo privileges (for service installation)
  • SSL certificate infrastructure (production deployments only)
⚠ Important: Verify all prerequisites are confirmed before proceeding. Do not advance to Section 3 until all boxes can be checked.

4. Installation Procedure

1
Install Redis Server

Redis must be installed and running before starting the application.

macOS:
brew install redis brew services start redis redis-cli ping # Expected: PONG
Ubuntu / Debian:
sudo apt-get install redis-server sudo systemctl start redis-server sudo systemctl enable redis-server redis-cli ping # Expected: PONG
2
Register MyCase OAuth Application
  1. Navigate to https://developer.mycase.com and sign in
  2. Select Create New Application
  3. Enter an application name and description
  4. Set the Redirect URI to match your intended REDIRECT_URI value (e.g. http://localhost:3000/api/auth/mycase/callback for development)
  5. Save the application and copy the Client ID and Client Secret to a secure location
3
Clone Repository & Install Dependencies
git clone <repository-url> cd mycase-oauth npm install
4
Configure Environment Variables

Create a .env file in the project root. Populate it with the values below, substituting placeholders with real credentials.

# Runtime Environment NODE_ENV=development PORT=3000 HOST=localhost # MyCase OAuth Credentials (from Step 2) MYCASE_CLIENT_ID=your_client_id MYCASE_CLIENT_SECRET=your_client_secret MYCASE_AUTH_URL=https://auth.mycase.com/login_sessions/new MYCASE_TOKEN_URL=https://auth.mycase.com/tokens # Application URLs REDIRECT_URI=http://localhost:3000/api/auth/mycase/callback BASE_URL=http://localhost:3000 # JWT Configuration (generate a strong random string) JWT_SECRET=your_jwt_secret_here JWT_EXPIRES_IN=8h # Redis Connection REDIS_URL=redis://localhost:6379 REDIS_DB=0 # Security ALLOWED_DOMAINS=localhost,127.0.0.1 # Logging LOG_LEVEL=info LOG_FORMAT=json # Token Storage TOKEN_ENCRYPTION_KEY=your_encryption_key_here USE_FILE_STORAGE=true # Admin Credentials (Optional) ADMIN_USERNAME=admin ADMIN_PASSWORD_HASH=bcrypt_hash_here ADMIN_JWT_SECRET=admin_jwt_secret_here

5. Starting the Server

Development Mode

npm run dev

Expected console output upon successful startup:

Production Mode

Verify Startup

curl http://localhost:3000/api/healthz # Expected: HTTP 200 with JSON health status

6. API Endpoint Reference

Authentication Endpoints

MethodEndpointDescription
GET/api/auth/mycaseInitiates OAuth 2.0 authorization flow — redirects to MyCase login
GET/api/auth/mycase/callbackOAuth callback handler — exchanges code for tokens
POST/api/auth/refreshRefreshes an expired access token using the stored refresh token
POST/api/auth/logoutRevokes tokens and clears session data

MyCase API Proxy Endpoints

MethodEndpointDescription
GET/api/mycase/*Proxies authenticated requests to the MyCase API
POST/api/mycase/*Proxies write operations to MyCase

Admin & Monitoring Endpoints

MethodEndpointDescription
GET/api/healthzHealth check — returns service status and Redis connectivity
GET/adminAdmin panel (requires admin credentials)
GET/api/admin/tokensLists all stored tokens (admin only)
DELETE/api/admin/tokens/:idRevokes a specific token (admin only)

8. Security Controls

ControlImplementation
Token encryptionAccess and refresh tokens are encrypted at rest in Redis using TOKEN_ENCRYPTION_KEY
JWT authenticationAll API requests require a valid JWT signed with JWT_SECRET
Domain allowlistRequests from domains not in ALLOWED_DOMAINS are rejected
Admin authenticationAdmin panel requires separate credentials; password stored as bcrypt hash
HTTPS enforcementProduction deployments must use HTTPS — configure via reverse proxy (Nginx/Caddy)
Secret isolationAll credentials are in .env — never hardcoded in source files

9. Production Deployment Checklist

Complete all items before go-live:

  • Node.js 18+ installed on the production server
  • Redis 6+ installed, running, and accessible
  • MyCase OAuth application registered with production Redirect URI
  • .env populated with production credentials (no placeholder values)
  • NODE_ENV=production set
  • Strong, unique values for JWT_SECRET, TOKEN_ENCRYPTION_KEY, and ADMIN_JWT_SECRET
  • HTTPS configured via reverse proxy
  • ALLOWED_DOMAINS restricted to production domains only
  • npm test passes with zero failures
  • npm audit shows no high/critical vulnerabilities
  • Health check endpoint returns 200: curl https://your-domain.com/api/healthz
  • OAuth test flow completes end-to-end in production environment

10. Common Issues & Resolutions

IssueLikely CauseResolution
Server fails to start — "Redis connection refused"Redis is not runningStart Redis: brew services start redis or sudo systemctl start redis-server. Verify with redis-cli ping.
OAuth callback returns 400 — "redirect_uri mismatch"Redirect URI in .env does not match what was registered in the MyCase Developer PortalUpdate the Redirect URI in the MyCase app registration to exactly match REDIRECT_URI in .env.
401 on all API proxy requestsJWT expired or invalidRe-authenticate via /api/auth/mycase. Check JWT_EXPIRES_IN setting.
Token not found in Redis after restartRedis was restarted without persistence enabledEnable Redis persistence (appendonly yes in redis.conf) or set USE_FILE_STORAGE=true for file-based fallback.
Admin panel returns 403Incorrect admin credentials or ADMIN_PASSWORD_HASH not setRegenerate the bcrypt hash and update ADMIN_PASSWORD_HASH in .env. Restart the server.
