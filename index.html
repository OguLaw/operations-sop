HubSpot Multi-Brand Rules

Oguz Law HubSpot account hosts multiple brands. Since separate companies share a single HubSpot portal, the following rules must be followed at all times.

Rule 1 — Brand Selection When Creating Assets

Before creating a new form, file, or any other asset, click the brand name in your profile area, then select the correct brand from the Brands section. Every asset must be created under the brand it belongs to.

Rule 2 — Access Management for Segments & Dashboards

Whenever you create a dashboard, segment, or other shared resource, go to Manage Sharing / Access and restrict access to only the teams relevant to that company.

For Oguz Law assets specifically, the following teams must NOT be given access:

  • EXTERNAL – ITB
  • EXTERNAL – IAU
  • Consultants
  • ITIN
  • Oguz Injury

Contact Page Views

What is a View?

A View is a saved filter with specific column arrangements. It lets you focus on certain contacts without reapplying filters each time. Access via CRM → Contacts → "All Views".

Creating a Custom View

  1. CRM → Contacts → Add View → "Create new view"
  2. Name the view; choose visibility (private / team / everyone)
  3. Add filters (lifecycle stage, lead status, last activity, owner, custom properties)
  4. Click "Save view"

Lead Management Process

Lead Creation Rules

  • Create the contact during or before the call so OpenPhone logs automatically
  • Can create with just a phone number — fill details after
  • Must enter email + phone number to prevent duplicate contacts

Four Required Questions on Initial Call

  1. What is your email address?
  2. What is your phone number? (double-check — pin additional numbers as notes)
  3. Where did you find us?
  4. Have you contacted our company before?

Lead Status Workflow

1-2-3-7 Day Follow-Up Method

When a call goes unanswered: set Lead Status to "Attempted to Contact" and create follow-up tasks for 1 day, 2 days, 3 days, and 7 days after the attempt. If still no answer: update to "No Answer".

Deal Process

  1. Create deal linked to the contact
  2. Set Deal Stage to "Contract Sent" (all deals must start here)
  3. Update Contact Lead Status to "Open Deal"
  4. Set up tasks and reminders for the contact owner
  5. If accepted: Deal Stage = Closed Won → Lead Status = Active Client
  6. If rejected: Deal Stage = Closed Lost → select Closed Lost Reason → Lead Status = Not Moving Forward → set Long-Term Follow-Up task

Referral Company Communication

Why This Approach Exists

Even with restricted HubSpot access, external referral reps could see all Notes, Calls, and Emails through Notes/Calls views — a confidentiality risk. The current approach solves this.

How It Works

  • Referral reps are added to HubSpot but not assigned to any team
  • When contact owner changes to a referral rep → automation creates a pinned note
  • Your team copies summaries into the comments section of that note and @mentions the rep
  • Reps can only see the pinned note and its comments — not the original activities

When a Referral Company Sends Us a Contact

  1. They create the contact (they become initial owner)
  2. Auto-pinned note appears — they add context in comments
  3. They set themselves as Secondary Contact Owner first
  4. Then they change Contact Owner to Hassan Kadak (Intake Leader)
⚠ If they forget to assign themselves as secondary owner before reassigning, they lose access. Remind them if needed.

WhatsApp UTM Tracking — SOP-ZAP-001

This workflow captures and stores UTM (Urchin Tracking Module) parameters from WhatsApp traffic via Zapier. It ensures that each unique visitor session — identified by a client ID (CID) — is recorded exactly once in the tracking table, preventing duplicate entries and maintaining clean attribution data.

SOP ID: SOP-ZAP-001  |  Status: Active  |  Platform: Zapier + Zapier Tables  |  Version: 1.1

1. Scope

This SOP applies to:

  • Marketing and analytics teams responsible for campaign attribution
  • Operations and automation teams managing Zapier workflows
  • Any stakeholder reviewing WhatsApp campaign performance data

2. Tools & Systems

ToolRole
ZapierWorkflow automation engine
Zapier WebhookReceives the incoming UTM payload (trigger)
Zapier Tables — WA_UTM_ClicksStores one record per unique CID

3. Trigger

The workflow is triggered by an incoming Webhook (HTTP POST). This webhook fires whenever a user arrives at a tracked page via a WhatsApp link containing UTM parameters.

Expected webhook payload fields:

FieldDescription
cidUnique client/session identifier
utm_sourceTraffic source (e.g. whatsapp, google)
utm_mediumMarketing medium (e.g. cpc, organic)
utm_campaignCampaign name
utm_termPaid search keyword (optional)
utm_contentAd variant (optional)
page_urlLanding page URL
referrerReferring URL
timestampISO 8601 timestamp of the session

4. Workflow Steps

The automation runs through four sequential steps each time the webhook fires:

1
Receive Webhook (Trigger)

The workflow starts when the Zapier webhook endpoint receives a POST request. The payload is parsed and all UTM fields are made available to subsequent steps. No action is taken at this point.

2
Search for Existing Record

Zapier searches the WA_UTM_Clicks table in Zapier Tables for the incoming CID.

  • Lookup field: cid (mapped to column f1)
  • Match type: exact match
  • If no match found → workflow continues to Step 3
  • If match found → filter in Step 3 will stop the workflow

Note: The search is configured to succeed on miss — it does not fail if no record is found; it continues normally.

3
Filter — Skip Duplicates

Condition: record_id from Step 2 does not exist.

  • If record_id is empty (no match) → proceed to Step 4, create new entry
  • If record_id has a value (match found) → workflow stops, no duplicate written
⚠ Deduplication guarantee: Each unique CID is stored only once, regardless of how many times the webhook fires for the same session.
4
Create Record in Zapier Tables

If no duplicate is found, a new row is written to WA_UTM_Clicks with all 9 UTM/page/timestamp fields mapped to the correct table columns.

5. Process Flow

Webhook fires ↓ Step 1: Parse payload (CID + UTM fields) ↓ Step 2: Search WA_UTM_Clicks for CID ↓ Step 3: Is record_id empty? YES → Step 4: Write new row NO → STOP (duplicate suppressed)

6. Testing Checklist

  • Send a test webhook payload with a new CID → confirm a record is created in WA_UTM_Clicks
  • Send the same CID a second time → confirm no duplicate record is created
  • Verify all 9 UTM/page/timestamp fields are correctly mapped to the right columns
  • Monitor Zapier task history for 48 hours and confirm no failures

7. Maintenance & Ownership

Review this Zap whenever:

  • WhatsApp campaign UTM parameter structure changes
  • The WA_UTM_Clicks table schema is modified
  • Zapier updates the Webhook or Tables integration

1. Executive Summary

We have built a two-Zap automated system in Zapier that tracks UTM attribution for inbound WhatsApp leads. The system captures the marketing source of a visitor before they send a WhatsApp message, then automatically applies that attribution data to their contact record in Wati when they first make contact.

Without this system, all WhatsApp leads appear as unattributed — making it impossible to determine which campaigns, ads, or channels are driving inbound enquiries.

2. System Architecture

The system consists of two Zapier workflows operating in sequence, using two Zapier Tables as shared data stores:

ComponentRole
Zap #1 (SOP-ZAP-001)Receives UTM parameters from the website via webhook when a visitor clicks a WhatsApp link. Stores UTMs + CID in WA_UTM_Clicks table.
Zap #2Fires when a new message arrives in Wati. Extracts the CID from the greeting, looks up UTM data in WA_UTM_Clicks, and writes attribution to the Contacts table and Wati contact attributes.
WA_UTM_Clicks tableStores one row per unique CID: UTM source, medium, campaign, term, content, landing page, referrer, timestamp.
Contacts tableStores attribution data tied to each Wati contact: CID, UTM fields, attribution timestamp, "already attributed" flag (f10).

3. End-to-End Flow

1. Visitor clicks WhatsApp button on website (link contains CID + UTM params) 2. Page script fires POST to Zapier webhook → Zap #1 stores UTMs in WA_UTM_Clicks 3. Visitor sends first WhatsApp message in Wati (greeting includes CID) 4. Zap #2 fires on new Wati message 5. Formatter step extracts CID from greeting text 6. Lookup: search WA_UTM_Clicks for that CID 7. If found: write UTM data to Contacts table + update Wati contact attributes 8. Set f10 = true (already attributed) → future messages from this contact are skipped

5. Sync Flow Details

  1. Lock acquisition — syncService uses a timestamp-based lock (LOCK_TIMEOUT_MS = 10 min). Stale locks are auto-released.
  2. Fetch hearings — caseService calls EOIR API with a 30 s timeout.
  3. Filter — only hearings within the next 12 months are kept; duplicates by ScheduleId are removed.
  4. Fetch details — parallel batches of 10 requests with up to 3 retries and exponential backoff (base 200 ms). 429 responses are respected.
  5. Read previous data — sheetService reads both individuals and others tabs (range A2:P).
  6. Diff & write — compares hearing date/time/judge/court, marks rows as NEW / UPDATED, preserves judge rates, notes, and assigned lawyer unless a change is detected in production mode.
  7. Trailing cleanup — rows beyond the new data (up to row 10,000) are cleared to prevent stale entries.
  8. Lock release — always executed in a finally block.
